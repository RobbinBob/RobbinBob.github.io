Before you can start making your custom trains or block scripts, there is a small bit of setup required to make sure everything can play nicely with each-other. For starters the NLVMCore types are part of a package called 'scripts', this means all custom type must also be in the same package so that they can access necessary types. Currently NLVM does not support walking up package heirarchy but I do hope soon that is changed, so for now keep it in the base 'scripts' package unless you know what you're doing! I'll highlight this here as it can help with some confusion that will happen along the line; Behaviour class *must* be attached to a SceneObject for it to work. They cannot be instanced in code and NLVM will make sure of that. If you want similar functionality that can be instanced at run-time then I suggest reading further down at different ways of receiving callbacks. For most purposes you'll want to extend Behaviour which mimics somewhat the native Script class with some extra QoL. If you're just going to skip the Behaviour class then you can skip along to another section as the next few chapters cover the basics of using Behaviours and their counterparts. The next chapter will cover receiving callbacks using Behaviours and other optional types.

All function begins with the Behaviour class, this replaces the native Script class with easier to use functionality. Just like with Script, Behaviour has a few update methods that can be overridden to add logic;
Start
Called once as the Behaviour is being initialized
- Update
Called each render update, passing in the time taken between the previous frame and now
- LateUpdate
Called at the end of each render update, passing in the time taken between the previous frame and now
- LateUnblockedUpdate
Called at the end of each render update, unaffected by simulation pause state
- Destroy
Called just before the Behaviour is stopped
Method callbacks differ slightly from Script type where LateUpdate will not be called when paused, this is to correct unintended behaviour of Script onLateUpdate which was kept for its use cases. If you wish to receive updates whilst the simulation is paused; use the LateUnblockedUpdate instead as this mimics onLateUpdate. There is also a lack of UnblockedUpdate which is due to only the late update method having the bug present, if this changes in future I will update the API.

For more complex use cases there is also an interface type which can recieve updates similar to FrameListener; IUpdateable. This interface exposes a couple methods that can receive callbacks when registered using the Behaviour register methods, allowing for more complex logic and removes the need for all update logic to be placed in Behaviours. It is important to note however that a Behaviour instance must be available to handle update requests which is a limitation of NLVM itself. 

                    
Behaviour.RegisterUpdateable(IUpdateable updateable);
Behaviour.RemoveUpdateable(IUpdateable updateable);

There is also a provided class type; UpdateableObject, that can be extended which removes just simplifies the registration to a single method, purely to reduce a small bit of code copying and removes a single extra line :0
